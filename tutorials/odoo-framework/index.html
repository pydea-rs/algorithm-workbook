Chapter 1Architecture Overview

Odoo is a modular, monolithic ORM-driven application framework — think Django Admin + Django REST Framework + a frontend framework, all tightly integrated. Everything is a module. You define a model, and Odoo auto-generates CRUD UI, API endpoints, security rules, and more.

Django / FastAPI / NestJS

You assemble models, views, serializers, admin, permissions, templates, and migrations as separate concerns. You wire them together explicitly.

Odoo

All-in-one: define a model → auto-generated CRUD UI, API, security, reports, website pages. Convention over configuration. A single model definition gives you a full-stack feature.

Request Lifecycle

  1. Browser/RPC sends request to Odoo server (port 8069)
  2. Web controller dispatches to appropriate handler (HTTP or JSON-RPC)
  3. Security layer checks ACL + record rules (like Django middleware)
  4. ORM processes business logic in Python (Active Record pattern)
  5. PostgreSQL executes queries (single database backend only)
  6. Response returns JSON / QWeb-rendered HTML / redirect
💡 Mental Model Think of Odoo as a "batteries-included" Django where the admin panel, REST API, template engine, and frontend are all one cohesive system. You spend less time wiring things together and more time on business logic.

  • Odoo is a modular monolith — all features live in modules
  • Models auto-generate UI, API, and security — no separate admin setup
  • Uses PostgreSQL only — leverages PG-specific features heavily
  • OWL is Odoo's custom frontend framework — similar to Vue but XML-based

⚠️ Data File Order Matters Load order is critical: security → data → views → menus. If a view references a model that doesn't exist yet, Odoo will crash during installation. Always put security files first.

Odoo's ORM is an Active Record implementation — similar to Laravel's Eloquent or Rails ActiveRecord. Unlike Django's Data Mapper pattern, Odoo models are tightly coupled to the UI layer. A single model definition gives you the database table, admin interface, API, and security hooks.

Model Types

from odoo import models, fields, api

# Regular model — creates a database table (like Django's Model)
class SaleOrder(models.Model):
    _name = 'sale.order'            # Technical name → table: sale_order
    _description = 'Sale Order'     # Human-readable name (REQUIRED in v14+)
    _inherit = ['mail.thread']       # Mixin: adds chatter functionality
    _order = 'date_order desc, id'  # Default sort (like Django's ordering)
    _rec_name = 'name'              # Field used for display name

    name = fields.Char(string='Order Reference', required=True, default='New')

# Abstract model — no DB table, pure behavior mixin
class MailThread(models.AbstractModel):
    _name = 'mail.thread'
    _description = 'Mail Thread'
    message_follower_ids = fields.One2many(...)

# TransientModel — auto-cleaned, for wizards (like Django sessions)
class SaleOrderWizard(models.TransientModel):
    _name = 'sale.order.wizard'
    _description = 'Sale Order Wizard'
    order_id = fields.Many2one('sale.order')

Inheritance — Extending Existing Models

This is one of Odoo's most powerful features. You can add fields/methods to any model from any module without touching the original code:

# Extend res.partner by adding a new field
class ResPartnerExtend(models.Model):
    _inherit = 'res.partner'   # No _name → extends in-place

    loyalty_score = fields.Float(string='Loyalty Score')
    preferred_currency_id = fields.Many2one('res.currency')

    def compute_loyalty_tier(self):
        for partner in self:
            if partner.loyalty_score > 100:
                partner.loyalty_tier = 'gold'
            else:
                partner.loyalty_tier = 'silver'

    loyalty_tier = fields.Char(compute='compute_loyalty_tier')

Computed Fields

Computed fields are calculated from other fields — like Django's @property but with optional database persistence:

    # Computed field — recalculated on every read
    full_name = fields.Char(
        compute='_compute_full_name',
        store=True,            # Store in DB (triggers recomputation on dependency change)
        index=True,            # Can index stored computed fields
    )

    @api.depends('first_name', 'last_name')   # Recompute when these change
    def _compute_full_name(self):
        for record in self:
            record.full_name = f"{record.first_name} {record.last_name}"

    # Inverse method — allow writing to computed field
    def _set_full_name(self):
        for record in self:
            if record.full_name:
                parts = record.full_name.split(' ', 1)
                record.first_name = parts[0]
                record.last_name = parts[1] if len(parts) > 1 else ''

    full_name = fields.Char(compute='_compute_full_name', inverse='_set_full_name')

    # Search method — custom search for computed field
    def _search_full_name(self, operator, value):
        if operator not in ('ilike', 'like'):
            return [('first_name', operator, value)]
        return ['|', ('first_name', operator, value), ('last_name', operator, value)]

@api.model_create_multi
def create(self, vals_list):
    for vals in vals_list:
        if not vals.get('name'):
            vals['name'] = self.env['ir.sequence'].next_by_code('my.model')

Read

# search() — returns a RecordSet (like Django's QuerySet)
partners = env['res.partner'].search([
    ('is_company', '=', True),
    ('name', 'ilike', 'acme'),
], order='name', limit=10, offset=0)

# browse() — get by ID(s) — returns RecordSet
partner = env['res.partner'].browse(42)
partner = env['res.partner'].browse([1, 2, 3])

# read() — returns list of dicts
data = partner.read(['name', 'email', 'is_company'])[0]
# {'name': 'Acme', 'email': 'a@acme.com', 'is_company': True, 'id': 42}

# search_read() — combined search + read (single RPC call — efficient!)
partners = env['res.partner'].search_read(
    [('is_company', '=', True)],
    ['name', 'email'],
    order='name',
    limit=10
)

# search_count() — count without loading records
count = env['res.partner'].search_count([('is_company', '=', True)])

Recordset Operations

# Filter — like Django's .filter() but with Python lambda
companies = partners.filtered(lambda p: p.is_company)

# Map — extract field values (like Django's .values_list())
names = partners.mapped('name')            # ['Acme', 'Beta', ...]

# Group — partition a recordset (no Django equivalent)

# Returns: {True: RecordSet, False: RecordSet}

# Set operations
all_partners = recordset_a | recordset_b    # Union
common = recordset_a & recordset_b          # Intersection
difference = recordset_a - recordset_b      # Difference

# Check if empty / get length
if not partners:           # Empty recordset is falsy

len(partners)              # Number of records
partners.ids               # List of database IDs [1, 2, 3]

ORM & CRUD Summary

  • Odoo uses Active Record — recordsets are always ordered sets
  • search_read() is preferred over search() + read() — single round-trip
  • Always iterate recordsets: for record in self: — never assume single record
  • mapped() supports nested traversal: records.mapped('partner_id.name')
  • Recordsets are falsy when empty — use if not records: to check
  • .ids gives you the list of database IDs from a recordset

Pattern: Override create with auto-naming

Like Django's save() override — auto-generate a sequence number when creating records:

Pattern: Write override with side effects

Trigger actions when specific fields change — like Django signals but inline:

def write(self, vals):
    if 'state' in vals:
        old_states = {r.id: r.state for r in self}
    result = super().write(vals)
    if 'state' in vals:
        for record in self:
            old = old_states.get(record.id)
            if old == 'draft' and record.state == 'confirmed':
                record.message_post(body="Order confirmed!")
                record._send_confirmation_email()
    return result

Process large datasets efficiently — like Django management commands but in crons:

@api.model
def _cron_process大批量(self):

        try:

Chapter 8Relations & Domains

Domains are Odoo's query filter syntax — used everywhere (search, security rules, defaults, computed fields). They're equivalent to Django's Q() objects but expressed as nested lists.

Domain Syntax

# Basic format: [(field, operator, value)]
domain = [('name', 'ilike', 'test')]

# Operators:
#   =        Equal               !=       Not equal
#   >        Greater than        <        Less than
#   >=       Greater or equal    <=       Less or equal
#   ilike    Case-insensitive contains (LIKE %value%)
#   like     Case-sensitive contains
#   in       In list             not in   Not in list
#   =        True/False          child_of Hierarchical child
#   parent_of Hierarchical parent

# AND (implicit — comma separated)
domain = [
    ('name', 'ilike', 'test'),
    ('active', '=', True),
    ('is_company', '=', True),
]

# OR (explicit operator)
domain = ['|',
    ('name', 'ilike', 'test'),
    ('email', 'ilike', 'test'),
]

# NOT
domain = ['!', ('active', '=', False)]

# Complex: (name contains 'test' AND active) OR (is_company=True AND country = US)
domain = ['|',
    '&', ('name', 'ilike', 'test'), ('active', '=', True),
    '&', ('is_company', '=', True), ('country_id.code', '=', 'US'),
]
🔗 Domain Operator Precedence Operators are applied right-to-left. The pattern is always: op, op, leaf, leaf. '|' applies to the next two elements. '&' is implicit (you rarely need it). '!' negates the next single element.

Common Patterns

# Related field traversal (dot notation)
orders = env['sale.order'].search([
    ('partner_id.is_company', '=', True),      # Follow FK chain
    ('partner_id.country_id.code', '=', 'US'),
])

# Dynamic time domains
from datetime import datetime, timedelta
last_week = datetime.now() - timedelta(days=7)
orders = env['sale.order'].search([
    ('date_order', '>=', last_week),
    ('state', '=', 'sale'),
])

# Null checks
partners = env['res.partner'].search([
    ('email', '=', False),       # email IS NULL
    ('phone', '!=', False),      # phone IS NOT NULL
])

# Context-based filtering
# Include archived records (normally filtered out)
orders = env['sale.order'].with_context(active_test=False).search([])

# Use context to pass dynamic values to search
env['sale.order'].with_context(default_partner_id=42).create({})

Chapter 9Inheritance Patterns

Odoo provides three distinct inheritance mechanisms — more than any other framework. Each serves a different purpose:

1. Classical Inheritance (_inherit + _name)

Creates a new model that inherits from an existing one. Like Python class inheritance — the new model gets all fields/methods. A new database table is created. This is what you use when you need a specialized version of an existing model.

class SaleOrderInherit(models.Model):
    _name = 'sale.order'                        # Same name as _inherit
    _inherit = ['sale.order', 'mail.thread']    # Inherit from existing models

    # Adds new field to sale.order
    priority = fields.Selection([('0', 'Normal'), ('1', 'High')])
    notes = fields.Text()

    def custom_method(self):
        # Has access to all sale.order methods
        for order in self:
            order.priority = '1'

2. Extension Inheritance (_inherit only, no _name)

Extends an existing model in-place — no new table. Like monkey-patching but clean. This is the most common pattern — used to add fields/methods to models from other modules (like adding loyalty_score to res.partner).

class ResPartnerExtend(models.Model):
    _inherit = 'res.partner'    # No _name → extends in-place

    # Adds field to existing res.partner
    loyalty_score = fields.Float(string='Loyalty Score')
    vip_date = fields.Date(string='VIP Since')

    def get_loyalty_discount(self):
        for partner in self:
            partner.vip_discount = 10.0 if partner.loyalty_score > 100 else 0.0

3. Delegation Inheritance (_inherits)

Composition pattern — model "has one" of another model. Fields from the delegated model are accessible directly, but it's a different table. Like Django's multi-table inheritance but more explicit. Use sparingly.

class AccountInvoice(models.Model):
    _name = 'account.invoice'
    _inherits = {'account.invoice.line': 'line_id'}
    # Fields from account.invoice.line are accessible directly
    line_id = fields.Many2one('account.invoice.line', required=True, ondelete='cascade')

    name = fields.Char()
    # Can access: record.product_id, record.quantity, etc.
    # (from account.invoice.line)
⚠️ Inheritance Rules
  • _inherit = string → _name is auto-set to the same value
  • _inherit = list → _name must be set explicitly (classical)
  • _inherits is for composition — methods are not inherited, only fields
  • Chained _inherits is not supported — avoid it

Chapter 10Constraints & Validation

from odoo.exceptions import ValidationError

class MyModel(models.Model):
    _name = 'my.model'

    name = fields.Char(required=True)
    amount = fields.Float()
    start_date = fields.Date()
    end_date = fields.Date()

    # Python constraint — like Django's clean() method
    # Runs when any of the listed fields change
    @api.constrains('start_date', 'end_date')
    def _check_dates(self):
        for record in self:
            if record.start_date and record.end_date:
                if record.start_date > record.end_date:
                    raise ValidationError("Start date must be before end date")

    @api.constrains('amount')
    def _check_amount(self):
        for record in self:
            if record.amount < 0:
                raise ValidationError("Amount must be positive!")

    # SQL constraints — enforced at database level (like Django's unique_together)
    _sql_constraints = [
        ('name_uniq', 'UNIQUE(name)', 'The name must be unique!'),
        ('amount_positive', 'CHECK(amount >= 0)', 'Amount must be positive!'),
        ('date_order_check',
         'CHECK(start_date <= end_date)',
         'Start date must be before end date!'),
    ]
💡 Python vs SQL Constraints SQL constraints are enforced at the database level — they always work, even for direct SQL. Python constraints run in the ORM layer — they give better error messages but can be bypassed by raw SQL. Use both for defense in depth.

id,name,model_id:id,group_id:id,perm_read,perm_write,perm_create,perm_unlink

🔐 ACL = First Gate ACL grants or denies entire operations (read/write/create/unlink). If ACL denies, record rules are never checked. ACL is like Django's Meta.permissions but simpler.

Record Rules

Record rules filter which records a user can see — like Django's object-level permissions or Row-Level Security:

<!-- Users can only see their own company's records -->
<record id="rule_my_model_company" model="ir.rule">
    <field name="name">My Model: Multi Company</field>
    <field name="model_id" ref="model_my_model"/>
    <field name="domain_force">[('company_id', 'in', company_ids)]</field>
    <field name="groups" eval="[(4, ref('base.group_user'))]"/>
</record>

<!-- Managers see ALL records -->
<record id="rule_my_model_manager" model="ir.rule">
    <field name="name">My Model: Manager Full Access</field>
    <field name="model_id" ref="model_my_model"/>
    <field name="domain_force">[(1, '=', 1)]</field>  <!-- Always true -->
    <field name="groups" eval="[(4, ref('group_manager'))]"/>
</record>

<!-- Users see only records they created -->
<record id="rule_my_model_personal" model="ir.rule">
    <field name="name">My Model: Personal Records</field>
    <field name="model_id" ref="model_my_model"/>
    <field name="domain_force">[('create_uid', '=', uid)]</field>
    <field name="groups" eval="[(4, ref('group_user'))]"/>
</record>
🔑 Record Rule Logic Rules for the same group are AND'd together. Rules for different groups are OR'd. A manager with both rule_company and rule_manager sees records matching EITHER rule.

@api.model
def _cron_cleanup_old_records(self):
    cutoff = fields.Datetime.now() - timedelta(days=30)
    old_records = self.search([
        ('create_date', '<', cutoff),
        ('state', '=', 'done'),
    ])
    old_records.unlink()

⚠️ Cron Best Practices
  • Batch your work — process records in groups of 100-1000
  • Use self.env['ir.cron']._commit_progress() for progress tracking
  • Don't call cron methods directly — let the scheduler run them
  • If a cron fails 5 times in 7 days, it gets auto-deactivated

  • Odoo's ORM is Active Record — recordsets are ordered sets, not lists
  • Always iterate with for record in self: — never assume single record
  • Use search_read() instead of search() + read() for efficiency
  • 3 inheritance patterns: classical (new model), extension (in-place), delegation (composition)
  • Security has two layers: ACL (operations) + Record Rules (rows)

  • SQL constraints = database level, Python constraints = ORM level
  • Use @api.depends for computed fields that should be stored in DB

from odoo.tests.common import TransactionCase, tagged
from odoo.exceptions import ValidationError

@tagged('post_install', '-at_install')
class TestMyModel(TransactionCase):

    def setUp(self):
        super().setUp()
        self.partner = self.env['res.partner'].create({
            'name': 'Test Partner',
            'email': 'test@example.com',
        })
        self.order = self.env['my.model'].create({
            'name': 'TEST-001',
            'partner_id': self.partner.id,
            'amount': 100.0,
        })

    def test_basic_creation(self):
        """Verify record creation"""
        self.assertEqual(self.order.name, 'TEST-001')
        self.assertEqual(self.order.partner_id, self.partner)
        self.assertTrue(self.order.active)

    def test_compute_fields(self):
        """Verify computed fields"""
        self.order.partner_id.first_name = 'John'
        self.order.partner_id.last_name = 'Doe'
        self.assertEqual(self.order.partner_id.full_name, 'John Doe')

    def test_constraint_negative_amount(self):
        """Verify validation raises error"""
        with self.assertRaises(ValidationError):
            self.order.write({'amount': -100})

    def test_search(self):
        """Verify search returns expected records"""
        orders = self.env['my.model'].search([
            ('partner_id', '=', self.partner.id)
        ])
        self.assertIn(self.order, orders)

    def test_workflow(self):
        """Test state transitions"""
        self.order.action_confirm()
        self.assertEqual(self.order.state, 'confirmed')
        self.order.action_done()
        self.assertEqual(self.order.state, 'done')

Chapter 24Best Practices

✅ Do

  • Always iterate recordsets: for record in self: — never assume single record
  • Use search_read() instead of search() + read() — single round-trip
  • Use @api.depends for computed fields that should be stored
  • Define _sql_constraints for data integrity at DB level
  • Use noupdate="1" for seed data users should customize
  • Write tests for every model — use TransactionCase
  • Keep controllers thin — business logic belongs in models
  • Use mapped() for efficient field access across recordsets
  • Use _description on every model (required since v14)
  • Batch operations instead of looping single records

❌ Don't

  • Don't use raw SQL unless absolutely necessary
  • Don't access self.env.uid in loops — cache it
  • Don't call create/write in loops — use batch methods
  • Don't use search() when you need IDs — use search_read()
  • Don't modify __manifest__.py data file order without care
  • Don't hardcode external IDs — always use ref()
  • Don't use super() without arguments in model methods
  • Don't put business logic in views/controllers
  • Don't skip _description — Odoo will warn
  • Don't forget to handle empty recordsets in computed fields
🏗️ Performance Pattern: Avoid N+1 
# ❌ Bad: N+1 queries (one query per record)
for order in orders:
    partner_name = order.partner_id.name

# ✅ Good: Prefetch in one query
partners = orders.mapped('partner_id')
# All partner data is now cached — no extra queries

# ❌ Bad: Individual creates
for data in data_list:
    Model.create(data)

# ✅ Good: Batch create
Model.create(data_list)  # Single SQL INSERT
